# List Competitive Alternatives

**Purpose:** Walk through Step 4 of the positioning process — identifying your true competitive alternatives as customers perceive them, not as you do. Use this command before you can meaningfully complete Step 5 (unique attributes) or any subsequent positioning steps.

**When to use:**
- You are starting or restarting a positioning exercise and have a best-fit customer profile in hand
- Your team disagrees on who your "real" competitors are
- Your current positioning feels generic because it was written without a clear alternative in mind
- You are preparing inputs for `/map-value-themes` or `/select-market-frame`

---

## Steps

1. **Confirm best-fit customer profile is available.**
   Read any positioning document, notes file, or user-supplied description to locate the best-fit customer definition. If none exists, prompt the user: "Before listing competitive alternatives, I need a best-fit customer profile. Run `/identify-best-fit-customers` first, or describe your best-fit customers now."

2. **Brainstorm all plausible alternatives.**
   For each best-fit customer segment identified, ask and answer the question:
   > "What would these customers do if our product didn't exist?"
   List every plausible answer — competing products, manual processes, spreadsheets, doing nothing, hiring consultants, building in-house, etc. Output this as a raw numbered list. Do not filter yet.

3. **Filter by customer mindshare.**
   Apply the inclusion decision rule to each item on the raw list:
   - **Include** if: a meaningful number of best-fit customers would actually use this alternative; it has mindshare among target prospects; it appears frequently when best-fit customers are asked "what would you do without us?"
   - **Exclude** if: the alternative has very few customers and therefore likely zero mindshare with prospects; only edge-case or atypical customers would consider it; it is only visible to experts with deep knowledge of the solution space, not to typical customers.

   Output the filtered list. Note excluded items in a separate "Edge cases (excluded)" section so the team can see what was removed and why.

4. **Rank filtered alternatives from most common to least common.**
   Ask the user: "Based on your customer conversations and sales data, how often does each alternative appear when best-fit customers describe what they'd use instead?" Use their input to sort the filtered list. If the user cannot rank with confidence, flag that customer interview data is needed before positioning can be finalized.

5. **Group into clusters.**
   Identify natural groupings among the ranked alternatives (e.g., "do it manually," "use a small-business accounting solution," "use a spreadsheet"). Aim for **two to five groups** — no fewer, no more. If you cannot reach two groups, ask the user whether the product truly has no distinct types of alternatives, as this may signal an overly narrow best-fit customer definition. If you exceed five groups, collapse the lowest-ranked ones or ask the user which edge cases to deprioritize.

6. **Output the final structured list.**
   Produce a table in this format:

   | Rank | Cluster Name | Alternatives in Cluster | Frequency Among Best-Fit Customers |
   |------|-------------|-------------------------|------------------------------------|
   | 1    | …           | …                       | Most common                        |
   | 2    | …           | …                       | …                                  |

   Label this output: **"Step 4 Output: Competitive Alternatives (input to Step 5)"**

   Offer the user the blank template at `../skills/obviously-awesome/templates/competitive-alternatives.md` — populate it directly with the generated table and edge-cases list, or hand it over for the user to fill in.

---

## Verify

The command succeeded when all of the following are true:

- [ ] The list was generated by asking "what would customers do without us?" — not by listing known industry competitors from a product-creator viewpoint.
- [ ] Every item on the final list passes the inclusion decision rule (meaningful mindshare, not edge-case-only).
- [ ] The list is ranked from most common to least common among best-fit customers.
- [ ] Alternatives are grouped into **2–5 clusters** with descriptive cluster names.
- [ ] The output table is explicitly labeled as input to Step 5.

If any check fails, surface the specific gap to the user before proceeding.

---

## Notes

**Anti-patterns to watch for:**
- If the team lists only named software vendors and omits "do nothing" or "manual process" as alternatives, the list is incomplete. Dunford is explicit that status quo and manual workarounds are often the most common competitive alternatives.
- If a competitor appears on the list solely because the product creator is aware of them — not because customers would actually use them — apply the **exclude** rule from Step 3. Low-mindshare competitors do not belong on this list.
